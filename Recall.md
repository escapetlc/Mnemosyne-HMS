# 记忆恢复 (Memory Restoration)

> 谟涅HMS 系统的记忆恢复层——通过人工粘贴记忆卡片 + AI辅助索引实现跨会话上下文恢复。  
> 本层准确名称是"记忆恢复"而非"语义召回"，因为它恢复状态（而非自动检索）。  
> 如需自动语义召回，建议部署 HMS 学术项目（[Shadow-Weave/HMS](https://github.com/Shadow-Weave/HMS)）作为后端。",


## 设计原则

**诚实定位**：HMS（Holographic Memory System）的召回层依赖向量数据库（pgvector）实现自动语义搜索，这是一个基础设施层组件。谟涅HMS 保持"协议原生、零基础设施"的定位，召回层等价物为"人工粘贴记忆卡片 + AI辅助索引"流程。如果未来需要自动语义召回，建议部署 HMS 学术项目（[Shadow-Weave/HMS](https://github.com/Shadow-Weave/HMS)）作为后端，本协议栈可作为其记忆提取与组织的前端。

---

## 适用规模与临界点

- **卡片数 < 10**：人工检索可行，系统运转良好
- **卡片数 10-50**：建议每积累 5-10 张跑一次 Dream 认知蒸馏做聚合，人工成本开始上升
- **卡片数 > 50**：人工关键词搜索的命中率将显著下降，"不记得位置"的概率陡增，记忆恢复层开始失效。此时建议引入向量化召回方案（部署 HMS 学术项目或类似系统），或接受恢复命中率下降

## 召回关键词（可选优化）

在提取记忆时，让 AI 同时生成一组"召回关键词"（5-10 个），附加在台账的 meta 字段中：

```
"recall_keywords": ["项目A", "决策冲突", "2026-07", "成本模型", "架构重构"]
```

这些关键词可作为文件名的扩展，或单独保存到标签索引文件中（`keywords_index.md`）。人工检索时多个维度，不完全依赖文件名。

## 召回流程

### Step 1：定位记忆卡片

当需要在新的对话窗口恢复上下文时：

- **如果你记得保存位置**：直接找到对应的 `谟涅台账_YYYYMMDD_主题.json` 或 `Dream归档_YYYYMMDD.md` 文件
- **如果不记得位置**：在本地按关键词搜索（`grep "关键词" ./\*` 或操作系统搜索功能）
- **如果有多张卡片**：优先选择时间最近的全量台账，再用旧卡片做补充

### Step 2：粘贴回忆

```
1. 打开任意 AI 对话窗口
2. 将记忆卡片内容复制粘贴进去
3. 输入：请读取以上记忆，确认我已恢复上下文
4. AI 阅读卡片后回复确认，此时可继续新任务
```

### Step 3（可选）：AI 辅助索引

当积累的记忆卡片超过 10 份时，使用**Dream认知维护协议**的"认知蒸馏模式"对多张卡片做一次跨卡聚合：

```
将多张记忆卡片粘贴进同一对话窗口，加载 Dream认知维护协议.json，
指定模式为"认知蒸馏"，AI 会输出：
- 跨卡的关键决策脉络
- 去重合并后的待办清单
- 跨项目元认知（如有多个项目）
```

输出结果可作为新的索引卡保存（`Dream索引_YYYYMMDD.md`）。

## 文件管理建议

```
谟涅HMS/
├── 台账/                  # 协议①产出的记忆卡片
│   ├── 谟涅台账_20260716_项目A.json
│   └── 谟涅台账_20260717_项目B.json
├── 归档/                  # 协议②产出的认知蒸馏
│   ├── Dream归档_20260716.md
│   └── Dream索引_20260716.md
└── 当前/                  # 当前活跃的记忆文件
    └── memory.md
```

## 与其他层的关系

```
对话文本 → ①谟涅Loci提取 → 记忆卡片（保存到台账/）
                              ↓ 人工粘贴
记忆卡片 → 新窗口粘贴 → 召回完成
                              ↓ 积累10+张
多张卡片 → ②Dream认知维护 → 跨卡聚合索引（保存到归档/）
                              ↓ 索引卡
索引卡 → 新窗口粘贴 → 全局上下文召回
```
