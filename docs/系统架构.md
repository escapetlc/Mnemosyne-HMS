# 谟涅HMS — 记忆服务系统

> **Mnemosyne Loci Holographic Memory System**  
> 协议原生的记忆提取、认知维护与跨会话记忆恢复系统。  
> 零基础设施，纯文本协议，跨任意 AI 窗口可执行。

---

## 系统概要

谟涅HMS 是一个基于纯文本 JSON 协议的轻量记忆服务系统，解决三类问题：

| 问题 | 对应组件 |
|------|----------|
| 对话结束后如何把状态带到新窗口 | **谟涅Loci记忆提取协议** (Retain) |
| 长期积累的记忆如何维护、去重、蒸馏 | **Dream认知维护协议** (Organize) |
| 需要历史记忆时如何找到并恢复 | **记忆恢复** (Recall.md) |

系统不依赖任何专有平台或运行环境。所有文件为纯文本 JSON/Markdown，复制粘贴到任意 AI 对话窗口即可执行。

**适用规模**：本系统最适合单人知识工作者使用，记忆卡片数控制在 50 份以内。10 份以下体验最佳，10-50 份需定期用 Dream 协议做认知蒸馏聚合。超过 50 份后人工搜索成本显著上升，建议引入向量化方案。

**【7·17 更新】** 根据三方评估报告调整：默认模式从全量改为坐标卡（日常使用优先轻量），压缩比从硬目标改为参考区间，5 型记忆中的 soul(灵魂自画像) 改为 AI 工作模型(operating_model)（只记录工作职责/输出规范/禁止行为，不记录人格描述）。协议新增规则9（弱模型精简执行）和卡片数临界点预警。

## 架构参考：HMS（Holographic Memory System）

本系统的架构设计深受 **HMS (Shadow-Weave Team)** 学术研究项目的启发。HMS 提出了一套"Retain → Recall → Organize → Answer"的完整记忆管线，以及基于证据台账(Evidence Ledger)的结构化记忆组织方法。

**区别**：HMS 是一个需要 PostgreSQL + API 服务器的基础设施级项目（[github.com/Shadow-Weave/HMS](https://github.com/Shadow-Weave/HMS)），而谟涅HMS 将同样的设计思想以纯文本协议的形式呈现，实现零部署、跨窗口可直接使用。

| HMS 学术概念 | 谟涅HMS 对应 |
|---|---|
| Retain（事实抽取） | 谟涅Loci记忆提取协议 |
| Organize（证据台账） | 五层空间坐标 + 事实类型分类 |
| fact-type: world/opinion/observation | 双语 fact-type: world(世界)/opinion(观点)/observation(观察) |
| Self-Evolution 4 控制 | R3 去重合并 / R7 源头索引 / 冲突仲裁 |
| bank_id 隔离 | Wing(工作区) 项目级别隔离 |

## 协议① vs 协议②：职责边界

| 维度 | ① 谟涅Loci记忆提取协议 | ② Dream认知维护协议 |
|------|------------------------|---------------------|
| 输入 | 单次对话文本 | 旧记忆文件 + 多份台账 |
| 输出 | 单次对话的结构化台账 | 更新后的长期记忆索引 / 跨卡蒸馏报告 |
| 使用频率 | 每次需要提取时 | 需要维护/聚合时（定期或阶段性） |
| 保真度 | 坐标卡(轻) / 全量(重) | 索引更新 / 认知蒸馏 |
| 生命周期角色 | 提取层（瞬态） | 维护层（长期） |

**简单记忆**：每次长对话结束后用 ① 提取，积累几份后用 ② 维护。

## 完整记忆生命周期

```
用户对话
    │
    ▼
① 谟涅Loci记忆提取协议
    │  模式：坐标卡 / 全量
    ▼
记忆台账（保存到本地 台账/ 目录）
    │
    ├──→ 粘贴到新窗口 → 上下文恢复（Recall）
    │
    └──→ 积累 3+ 份后
            │
            ▼
    ② Dream认知维护协议
        │  模式：索引更新 / 认知蒸馏
        ▼
    更新后的记忆索引 / 跨卡蒸馏报告（保存到 归档/ 目录）
            │
            └──→ 粘贴到新窗口 → 全局上下文恢复
```

## 使用方式

### 无需任何工具（推荐）

所有文件复制粘贴到任意 AI 对话窗口即可：

```
1. 打开任意 AI 对话窗口（Claude / GPT / DeepSeek / 千问 / Kimi 等）
2. 将协议 JSON 文件内容粘贴进去
3. （可选）将对话文本或记忆文件粘贴在后面
4. 输入指令：请执行此协议
5. AI 读取 JSON，按规则执行
```

**示例**：
```
用户："请执行此协议"
      （已粘贴谟涅Loci记忆提取协议.json + 一段对话文本）
AI：  "已读取协议。已识别 3 个话题，提取坐标卡如下：..."
```

## 术语表

| 术语 | 定义 |
|------|------|
| 谟涅 (Mnemosyne) | 希腊神话中的记忆女神，本系统的命名来源 |
| Loci (记忆宫殿法) | 一种将信息与空间位置关联的记忆方法，本系统的五层空间坐标法由此得名 |
| Dream | 项目级认知资产蒸馏的代号，来源于最初的跨窗口记忆提取实验命名 |
| 五层空间坐标 | Wing(工作区)/Room(房间)/Hall(厅)/Closet(柜)/Drawer(抽屉) 分层记忆模型 |
| 证据台账 (Evidence Ledger) | 在生成回答前将检索到的记忆组织为结构化证据清单的设计方法，源自 HMS 学术项目 |
| fact-type | 事实类型标签。本系统使用中英双语：world(世界)/opinion(观点)/observation(观察)/decision(决策)/pending(待办)/hazard(警示) |
| HMS (Holographic Memory System) | Shadow-Weave 团队开源的长期记忆问答框架，本系统的架构灵感来源 |
| 5 型记忆索引 | 记忆维护中使用的五种分类：operating_model(AI工作模型)/user(服务对象)/feedback(纪律红线)/project(当前任务)/reference(技术资产) |
| 坐标卡 | 谟涅Loci协议的轻量输出模式，约 170 token，用于快速窗口状态恢复 |
| 全量台账 | 谟涅Loci协议的重型输出模式，完整五层空间 JSON，参考压缩比 85-95%，信息密度异常对话可豁免 |

## 继承关系

本系统取代以下旧协议。旧文件保留不动，新用户请使用本系统的文件。

| 旧协议 | 被取代方式 |
|--------|-----------|
| 谟涅记忆恢复 v2.0 | 并入库涅Loci协议，作为"坐标卡"模式 |
| AI对话复位压缩机 v1.1 | 并入库涅Loci协议，作为"全量"模式 |
| 记忆维护器 v1.1 | 并入库Dream协议，作为"索引更新"模式 |
| Dream认知器 v1.0 | 并入库Dream协议，作为"认知蒸馏"模式 |

**生态内互补文件（非本系统直接取代）：**
- `CLAUDE.md` — 通用AI协作角色与纪律（已纳入本目录）
- `Memory-Mode.md` — 5 型记忆索引空白模版（与Dream协议输出结构一致，已纳入本目录）
- `LIB-026 制成师知识库扩展模版` — 角色级知识补充，与本系统互补
- `KIMI额度计算协议.json` — 独立的统计计算协议，与本系统无直接关系
